# Agent 质量验收

[English](../AGENT_QUALITY_ACCEPTANCE.md) · [中文文档首页](../README.md)

自动断言通过只能证明路径可行，不能证明产品体验合格。真实 Agent 验收应基于 trace 和产物，把以下维度评为 `pass`、`warning` 或 `fail`。

## 六个维度

1. **任务完成**：用户目标端到端完成、持久化，并能在对话结束后查看。
2. **任务正确**：产物、提示词、参考、依赖、模板和导出满足所有关键约束。
3. **成本控制**：LLM 调用、工具调用、Token、缓存、耗时和重复读取与任务复杂度匹配。
4. **灵活性**：相近输入、局部修改、模板复用和不同 Skill 不依赖硬编码短语或固定模板 ID。
5. **用户体验**：只询问阻塞问题，进度清楚，不把内部账本和工具细节当成用户回复。
6. **可观测性与安全**：trace、工具结果、节点状态、Token、产物、失败、重试和破坏性确认互相一致。

## 验收证据

报告至少包含：

- 用户原始目标和完成条件；
- 最终持久化的节点、资产、工作流或导出；
- 关键工具和 trace 事件；
- Token、缓存和耗时；
- 用户界面改动的真实截图；
- 失败与重试原因；
- 破坏性操作的确认记录。

如果任务完成和正确性通过，但成本、灵活性或用户体验仍是 warning，结论只能是：

```text
Feasible, but not yet good enough.
```

只有全部维度通过，或版本明确接受剩余 warning，才能宣称达到生产质量。
