# 工作流指南

[English](../en/workflows.md) · [中文文档首页](../README.md)

## 什么时候使用工作流

普通的单次图片或视频任务可以直接创建节点。以下情况更适合工作流：

- 相同生产步骤会在多个项目中重复使用；
- 需要明确输入、依赖、集合和循环；
- 需要让团队成员按同一流程执行；
- 需要把“准备提示词”和“真正调用媒体模型”分开；
- 需要查看每个步骤的运行状态和失败位置。

## 两张画布

- **流程面板**描述可复用的步骤结构。
- **创作画布**展示每次运行产生的文本、图片、视频和音频。

流程中的集合整理、循环规划和输入解析可以只保存在运行时；用户真正需要查看和修改的产物才创建画布节点。

## 使用现有模板

1. 打开流程面板。
2. 从内置或“我的”模板中选择流程。
3. 阅读模板说明和必填输入。
4. 填入故事、时长、画幅、素材或其他参数。
5. 选择运行下一步、指定步骤或全部步骤。
6. 回到创作画布检查生成的节点和参考关系。

内置模板保留一份可恢复来源。修改内置模板时，应在用户空间保存可编辑副本；重置会恢复内置定义，不覆盖你的其他用户模板。

## 搭建工作流

进入 Workflow Build Mode 后，从输入和最终产物反向设计：

1. 定义运行前必须提供的 `inputs`。
2. 添加文本规划或内容生成步骤。
3. 只有数量确实动态时才使用集合和循环。
4. 为每一步设置真实执行依赖。
5. 为图片和视频步骤指定需要的视觉参考选择器。
6. 标记哪些步骤创建画布产物，哪些只保留运行时状态。
7. 使用画布投影检查布局和依赖，无需先运行媒体模型。
8. 保存为用户模板后，用一组不同输入做完整测试。

## Prompt 的职责

每个步骤的提示词只描述当前步骤需要完成的任务，不应该把整个工作流 JSON、所有节点详情或无关历史塞给模型。推荐结构：

- 角色或专业身份；
- 当前步骤任务；
- 可用输入和上游结果；
- 输出要求；
- 自检标准。

内容提示词输出正常文本。只有步骤协议明确要求结构化数据时才输出结构化结果，不要在人物设定、场景描述或最终视频提示词中无理由输出 JSON。

## 依赖与参考

- `needs` 表示步骤必须在上游完成后才能执行。
- 节点 `fields.references` 表示产物实际读取哪些文本或视觉素材。
- `parent_node_id` 只负责画布分组，不代表生产依赖。
- 视频节点通常同时参考人物、场景和分镜；参考选择器可以从集合中挑出当前段真正出现的素材。

执行顺序和画布位置是两件事。不要为了排版制造假的依赖，也不要因为节点放在旁边就省略真实依赖。

## 集合与循环

当人物数、分段数或分集数来自输入或上游输出时，循环必须有明确来源：

- `for_each` 指向数组；
- 或 `repeat.count` 指向确定数量；
- 或使用协议支持的分集数与分段数组合。

循环子步骤读取当前 item，并通过稳定模板步骤 ID 生成不同运行实例。不要复制多份“从第一个元素开始”的固定步骤来模拟循环。

## 手动生成

手动生成不是跳过步骤。流程仍然会：

1. 收集上游结果；
2. 创建目标节点；
3. 写入最终提示词和参考关系；
4. 停在调用媒体模型之前。

用户确认后再运行该节点。

## 保存与版本

- 修改自己的模板时更新原模板。
- 从内置模板开始修改时保存一个用户副本。
- “另存为”用于明确创建新模板，而不是每次保存都自动复制。
- 导入前检查 `required_capabilities` 和 `required_extensions`。
- 缺失必需扩展时应在保存或运行前报错。

## 发布前自查

- 输入名称和类型清晰；
- 所有步骤 ID 唯一且稳定；
- 没有缺失或循环依赖；
- 每个循环都有确定数据源；
- 图片和视频参考完整；
- 画布产物与运行时中间态区分正确；
- 手动步骤停在媒体调用前；
- 用至少一组真实输入完成画布投影和运行测试。

底层字段见 [Workflow Spec 协议](./workflow-spec-protocol.md)。
